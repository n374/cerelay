# F4-cross-cwd-fileproxy-isolation 设计文档
## 1. 架构决定
当前仓库中 e2e 目录实际是 `test/e2e-comprehensive/`，不是需求写的 `server/test/e2e-comprehensive/`；现有 F4 用例位于 `test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1098`，且注释明确说明它只守 `sessionId` 唯一、`spawn.ready detail.cwd` 对齐、同 client 共享 `deviceId`，不守 fileProxy、FileAgent、ancestor walk、project-claude bind mount 路径（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1085`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1090`）。
### 不变量 a：fileProxy 三 root 内容隔离
守护点应是 namespace 内通过 `serverExec.run` 主动读取三类根对应路径，而不是读取 client 本地文件。`serverExec.run` 是当前 honest 触发 namespace 内 FUSE read/write 的入口，helper 注释已经说明 client-routed bash 不进入 namespace，必须用 `/admin/sessions/:id/exec`（`test/e2e-comprehensive/orchestrator/server-events.ts:447`、`test/e2e-comprehensive/orchestrator/server-events.ts:451`、`server/src/server.ts:554`、`server/src/server.ts:571`）。因此断言应包含两层：命令 stdout 必须只含本 session marker；同时 admin event 必须证明该路径由 fileProxy 服务过。
`home-claude` 与 `home-claude-json` 现有 `file-proxy.read.served` 可作为正向证据，因为 snapshot-cache、snapshot-client、runtime cache、settings passthrough 都会记录该事件（`server/src/file-proxy-manager.ts:968`、`server/src/file-proxy-manager.ts:1181`、`server/src/file-proxy-manager.ts:1255`、`server/src/file-proxy-manager.ts:1911`）。但现有事件 detail 只有 `root`、`relPath`、`servedFrom` 等字段，缺少 `clientCwd` 与内容摘要，helper 也只按 `root`、`relPath`、`sessionId`、`since` 过滤（`test/e2e-comprehensive/orchestrator/server-events.ts:96`、`test/e2e-comprehensive/orchestrator/server-events.ts:163`）。为了做跨 cwd 负断言，需要扩展 read event detail，至少加入 `clientCwd`、`clientPath`、可选 `sha256` 或 `contentSampleHash`。
`project-claude` 的普通文件不走 cache，启动 snapshot 会从 client 拉 `project-claude` 并 emit `snapshot-client`（`server/src/file-proxy-manager.ts:869`、`server/src/file-proxy-manager.ts:871`、`server/src/file-proxy-manager.ts:890`、`server/src/file-proxy-manager.ts:968`）。但 `settings.local.json` 是 shadow file，daemon 本地读会通过 sideband emit `file-proxy.shadow.served`（`server/src/fuse-host-script.ts:676`、`server/src/fuse-host-script.ts:688`）。实际代码中 sideband 到 server 后由 `FileProxyManager` 用 `this.sessionId` 记录，并非需求中所说 `sessionId === null`（`server/src/file-proxy-manager.ts:1571`、`server/src/file-proxy-manager.ts:1577`）。因此本设计选择“修正/扩展 sideband detail”，不是绕开 shadow 事件；新增 detail 应包含 `clientCwd`、`fusePath`、`shadowPath`，用于区分两个并发 session。
### 不变量 b：FileAgent per-device singleton cache hit 污染
风险来源真实存在：server 按 `deviceId` 维护 `fileAgents` 单例池（`server/src/server.ts:108`、`server/src/server.ts:859`、`server/src/server.ts:900`），同一 `deviceId` 多 session 会共享 `FileAgent`（`server/src/server.ts:853`、`server/src/server.ts:854`）。同时 `ClientCacheStore` 说明 manifest 与 blob 池是 per-device，不再按 cwd 分区（`server/src/file-agent/store.ts:1`、`server/src/file-agent/store.ts:11`），`FileAgent.read` 命中也只查 `(deviceId, scope, relPath)`（`server/src/file-agent/index.ts:105`、`server/src/file-agent/index.ts:107`）。
这个不变量不能只查 `/admin/cache`，因为那会绕过 FUSE 与 FileAgent 相关运行路径。应通过 session A 先促成同 relPath 的缓存/预热，再让 session B 在 namespace 内读取同 relPath，并断言 B 的 stdout 不含 A marker，且 `file-proxy.read.served` 对 B 的 event 记录 `sessionId=B`、`clientCwd=cwdB`、`root=home-claude` 或 `home-claude-json`。`FileProxyManager` 运行时 cache hit 会在 `tryServeReadFromCache` 里命中 `cacheStore.lookupEntry(deviceId, scope, relPath)` 并记录 `servedFrom:"cache"`（`server/src/file-proxy-manager.ts:1205`、`server/src/file-proxy-manager.ts:1227`、`server/src/file-proxy-manager.ts:1255`），这是最接近污染风险的观测点。
需要新增一个 probe event 或扩展现有 event 来标记 cache lookup 的分区输入。建议扩展 `file-proxy.read.served`：加入 `clientCwd`、`cacheScope`、`cacheRelPath`、`clientPath`、`servedFrom`、可选 `sha256`。现有 helper 没有 negative assert，也没有 cwd filter（`test/e2e-comprehensive/orchestrator/server-events.ts:158`、`test/e2e-comprehensive/orchestrator/server-events.ts:182`），因此 orchestrator 需要新增“在 since 后没有匹配另一 cwd/marker/hash 的事件”的查询式断言，而不是只等正向事件。
### 不变量 c：cwd-ancestor walk 不得读到另一个 session 的 cwd 子树
实际代码中 `computeAncestorChain(cwd, homeDir)` 从 cwd 开始逐级向上直到 homeDir，不含 homeDir 和 fs root（`server/src/path-utils.ts:17`、`server/src/path-utils.ts:19`、`server/src/path-utils.ts:24`、`server/src/path-utils.ts:27`）。`ConfigPreloader` 会用该链构造 `CLAUDE.md` 与 `CLAUDE.local.md` 预热项（`server/src/config-preloader.ts:131`、`server/src/config-preloader.ts:133`），runtime env 也会把同一链写入 `CERELAY_ANCESTOR_DIRS`（`server/src/claude-session-runtime.ts:250`、`server/src/claude-session-runtime.ts:252`）。
但我发现一个与规格不一致的点：`FileProxyManager` 构造的 `this.roots` 只有 `home-claude`、`home-claude-json`、`project-claude`，没有 `cwd-ancestor-N`（`server/src/file-proxy-manager.ts:258`、`server/src/file-proxy-manager.ts:261`）。daemon 支持 `cwd-ancestor-*` 语义（`server/src/fuse-host-script.ts:451`、`server/src/fuse-host-script.ts:568`、`server/src/fuse-host-script.ts:634`），bootstrap 也会尝试从 `$CERELAY_FUSE_ROOT/cwd-ancestor-N/CLAUDE.md` bind mount（`server/src/claude-session-runtime.ts:331`、`server/src/claude-session-runtime.ts:348`），但 server 目前没有把这些 root 传给 daemon（`server/src/file-proxy-manager.ts:685`）。所以“通过真实 ancestor FUSE root 读取 CLAUDE.md”在现状下不可达。
本设计对不变量 c 分两级守护：第一层新增纯观测 probe event，在 `ConfigPreloader` 或 runtime env 构建时记录 `sessionId`、`clientCwd`、`ancestorDirs`、`prefetchAbsPaths`，断言 A 的 ancestor 列表不含 B cwd 子树，B 反之；这不改核心读逻辑，只是暴露已有计算结果。第二层如果后续补齐 `cwd-ancestor-N` root，则再用 `serverExec.run` 读取 namespace 内 ancestor `CLAUDE.md`，并用 `file-proxy.read.served root=cwd-ancestor-N` 断言路径真实经过 FUSE；在当前代码下，这部分应在设计里标为“需要产品/基础设施先补齐 root 映射”。
### 不变量 d：project-claude bind mount 严格来自本 session cwd
`project-claude` root 在 `FileProxyManager` 构造时固定为 `path.join(this.clientCwd, ".claude")`（`server/src/file-proxy-manager.ts:246`、`server/src/file-proxy-manager.ts:258`、`server/src/file-proxy-manager.ts:261`）。runtime bootstrap 会把 `$CERELAY_FUSE_ROOT/project-claude` bind 到 `$CERELAY_WORK_DIR/.claude`（`server/src/claude-session-runtime.ts:327`、`server/src/claude-session-runtime.ts:328`、`server/src/claude-session-runtime.ts:329`），而 `CERELAY_WORK_DIR` 来自 `options.cwd`（`server/src/claude-session-runtime.ts:250`）。
守护方式应是：两个 session 的 cwd 下分别放 `.claude/project-marker.txt`，并发启动后用 `serverExec.run(sessionA)` 读取 `cwdA/.claude/project-marker.txt`，用 `serverExec.run(sessionB)` 读取 `cwdB/.claude/project-marker.txt`。stdout 必须只包含本 cwd marker；同时 `file-proxy.read.served` 或 `file-proxy.client.requested` 必须记录 `root=project-claude`、`clientCwd` 等于对应 session cwd。`project-claude` 不是 cache scope，`rootToCacheScope` 对它返回 null（`server/src/file-proxy-manager.ts:2038`、`server/src/file-proxy-manager.ts:2044`、`server/src/file-proxy-manager.ts:2045`），所以它天然适合测试 bind mount/session runtime 是否串台，而不是 cache 污染。
该不变量需要扩展事件 detail，但不需要改 bind mount 核心逻辑。现有 `pty.spawn.ready` 只证明 PTY helper spawn 成功，detail 包含 `cwd` 和 `homeDir`（`server/src/pty-session.ts:211`、`server/src/pty-session.ts:212`、`test/e2e-comprehensive/orchestrator/server-events.ts:349`），不足以证明 `$cwd/.claude` 实际绑定到了本 session 的 FUSE `project-claude`。因此主断言必须读取 namespace 文件并配合 fileProxy event，而不是停留在 `spawn.ready detail.cwd`。
## 2. 基础设施改动清单
### server 侧
`server/src/admin-events.ts`：保留顶层 `sessionId` 结构，因为实际 schema 已支持 `sessionId: string | null`（`server/src/admin-events.ts:8`、`server/src/admin-events.ts:12`）。需要的不是改顶层结构，而是扩展各类 fileProxy/config probe 的 `detail` 字段；`AdminEventBuffer.record` 已接受任意 `Record<string, unknown>`，可直接承载新字段（`server/src/admin-events.ts:25`、`server/src/admin-events.ts:32`）。
`server/src/file-proxy-manager.ts`：扩展 `file-proxy.read.served`、`file-proxy.client.requested`、`file-proxy.shadow.served` 转录 detail。read served 的现有出口包括 snapshot-client（`server/src/file-proxy-manager.ts:968`）、snapshot-cache（`server/src/file-proxy-manager.ts:1181`）、runtime cache（`server/src/file-proxy-manager.ts:1255`）、settings passthrough（`server/src/file-proxy-manager.ts:1911`）；这些出口都应统一带上 `clientCwd`、`clientRoot` 或 `clientPath`、`root`、`relPath`、`servedFrom`。client requested 出口在穿透 client 前记录（`server/src/file-proxy-manager.ts:1699`、`server/src/file-proxy-manager.ts:1703`），也应带上 `clientCwd` 与 `clientPath`，否则无法证明 session B 没有请求 session A 的 cwd 子树。
`server/src/file-proxy-manager.ts` 的 sideband 转录路径目前把 daemon event 用 `this.sessionId` 记录（`server/src/file-proxy-manager.ts:1571`、`server/src/file-proxy-manager.ts:1577`），所以无需“修 sessionId null”这一项；但需要在转录时补充 `clientCwd`，或者要求 daemon detail 自带 `fusePath` 后由 server 补充 cwd。这个方向只增加观测字段，不改变 FUSE read/write 决策路径。
`server/src/config-preloader.ts`：新增 `config-preloader.plan` 或 `cwd-ancestor.plan` admin event，记录 `sessionId`、`homeDir`、`cwd`、`ancestorDirs`、`prefetchAbsPaths`。`ConfigPreloader` 已经集中计算 `ancestorChain` 和 prefetch items（`server/src/config-preloader.ts:101`、`server/src/config-preloader.ts:116`、`server/src/config-preloader.ts:131`），在这里加 probe 能直接守“不把另一个 cwd 子树加入预热计划”。
`server/src/claude-session-runtime.ts`：新增 `namespace.mount-plan` admin event 或在 server 调用处记录 runtime mount env 的关键字段。`buildMountNamespaceEnv` 已设置 `CERELAY_WORK_DIR`、`CERELAY_VIEW_ROOTS`、`CERELAY_ANCESTOR_DIRS`、`CERELAY_FUSE_ROOT`（`server/src/claude-session-runtime.ts:247`、`server/src/claude-session-runtime.ts:252`、`server/src/claude-session-runtime.ts:256`），但该函数当前不接收 `adminEvents`。更低风险的实现是在 `server/src/server.ts` 调用 `createClaudeSessionRuntime` 前记录 `message.cwd`、`fileProxy.mountPoint`、预期 project bind 目标；调用点在 `server/src/server.ts:1204` 至 `server/src/server.ts:1210`。
`server/src/server.ts`：在 session 创建期间把 `sessionId`、`deviceId`、`message.cwd`、`runtimeRoot`、`fileProxy.mountPoint` 作为 probe 记录。`runtimeRoot` 是按 sessionId 生成并清理的（`server/src/server.ts:1110`、`server/src/server.ts:1111`），`FileProxyManager` 创建时传入 `clientCwd` 与 `sessionId`（`server/src/server.ts:1145`、`server/src/server.ts:1148`、`server/src/server.ts:1149`）。这些字段能帮助 reviewer 判断 race/state leak 是否发生在 session 创建阶段。
### orchestrator 侧
`test/e2e-comprehensive/orchestrator/server-events.ts`：扩展 TypeScript detail interface。`FileProxyReadServedDetail` 当前没有 cwd、clientPath、hash 字段（`test/e2e-comprehensive/orchestrator/server-events.ts:96`、`test/e2e-comprehensive/orchestrator/server-events.ts:99`），`FileProxyShadowServedDetail` 当前没有 cwd（`test/e2e-comprehensive/orchestrator/server-events.ts:111`、`test/e2e-comprehensive/orchestrator/server-events.ts:115`），`FileProxyClientRequestedDetail` 当前也没有 cwd/clientPath（`test/e2e-comprehensive/orchestrator/server-events.ts:139`、`test/e2e-comprehensive/orchestrator/server-events.ts:143`）。需要补齐字段并让 helper 支持 `clientCwd` 过滤。
同文件还需要新增 negative/assert helper。现有 `waitForReadServed`、`waitForShadowServed` 都是等正向事件（`test/e2e-comprehensive/orchestrator/server-events.ts:182`、`test/e2e-comprehensive/orchestrator/server-events.ts:228`），B5 是直接 `findClientRequested` 后手写 length 为 0（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:346`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:348`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:353`）。F4-cross-cwd 需要可复用的 “find unexpected events since baseline where sessionId=B but clientCwd=cwdA/clientPath startsWith cwdA” 查询，失败时打印完整 fileProxy/config probe 摘要。
`test/e2e-comprehensive/orchestrator/phase-p1.test.ts` 或新增 `phase-p2.test.ts`：新增 P2 case。现有 P1 文件已经很长，且 F4 P1 注释明确 P2 加固未展开（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1090`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1094`），建议放到独立 `phase-p2.test.ts` 或 P2 分组文件，避免改变 P0/P1 已绿用例。
`test/e2e-comprehensive/orchestrator/fixtures.ts`：现有 `writeFixture` 能写共享 fixture cwd 子树（`test/e2e-comprehensive/orchestrator/fixtures.ts:8`、`test/e2e-comprehensive/orchestrator/fixtures.ts:15`），足够布置 `cwdA/.claude`、`cwdB/.claude`、父级 `CLAUDE.md`。home 侧 fixture 通过 `RunRequest.homeFixture` 写入 client `$HOME`（`test/e2e-comprehensive/orchestrator/clients.ts:7`、`test/e2e-comprehensive/agent/index.ts:211`、`test/e2e-comprehensive/agent/index.ts:218`），但同 client 并发两 session 共享同一个 `$HOME`，不能用它表达“不同 cwd 的 home root 内容不同”。因此 home root 隔离只能守“同 relPath 不被另一 session 的 cwd 污染”，不能构造同 device 同 home 下不同内容。
### daemon 侧
`server/src/fuse-host-script.ts`：sideband `emit_event` 当前只发送 `{type, kind, detail}`，没有 sessionId，也没有 cwd（`server/src/fuse-host-script.ts:388`、`server/src/fuse-host-script.ts:395`）。实际 server 已在转录时补上顶层 sessionId（`server/src/file-proxy-manager.ts:1577`），所以 daemon 不必承担 sessionId 绑定；但可在 shadow read/write detail 中加入 `fusePath`，例如 `path` 解析前的原始 FUSE 路径。shadow read 现有 detail 有 `root`、`relPath`、`shadowPath`、`bytes`（`server/src/fuse-host-script.ts:687`、`server/src/fuse-host-script.ts:688`、`server/src/fuse-host-script.ts:692`），补 `fusePath` 能定位是否来自 `$cwd/.claude` bind。
daemon 的普通 read 当前不 emit served event；普通 read 通过 `send_request` 进入 server（`server/src/fuse-host-script.ts:713`、`server/src/fuse-host-script.ts:714`），由 server 端现有 read/client requested event 观测即可。不要为了测试在 daemon 普通 read 上新增内容判定逻辑，否则会复制核心路径判断，降低测试价值。
daemon 的 `cwd-ancestor-*` 支持已经存在（`server/src/fuse-host-script.ts:451`、`server/src/fuse-host-script.ts:568`、`server/src/fuse-host-script.ts:634`），但 server 没有传对应 root（`server/src/file-proxy-manager.ts:258`、`server/src/file-proxy-manager.ts:685`）。本 P2 设计不建议在测试 PR 中顺手补这个核心行为；应作为单独产品/infra 问题处理，P2 case 先用 mount/preload probe 守计算不串台。
## 3. Case 设计
建议做 1 个综合 case，而不是 4 个完全独立 case。四个不变量共享同一拓扑：同 `client-a`、同 `deviceId`、两个并发 `runAsync`、不同 cwd、通过 `serverExec.run` 在各自 namespace 内触发读取。现有 F4 P1 已证明同 client 并发两个 child 会共享 deviceId（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1119`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1165`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1170`），P2 应复用该拓扑并加深 fileProxy 断言，避免四个 case 反复启动昂贵 PTY/FUSE。
fixture 布局建议使用一个父目录下两个兄弟 cwd：`case-f4-cross/a` 与 `case-f4-cross/b`。每个 cwd 下放 `.claude/project-marker.txt`，内容分别是 `PROJECT_A_ONLY`、`PROJECT_B_ONLY`，用于不变量 d。父级放 `CLAUDE.md`，两个 cwd 内也可放本地 `CLAUDE.md`，内容分别是 `ANCESTOR_A_ONLY`、`ANCESTOR_B_ONLY`，用于验证 ancestor plan 不跨到兄弟目录。`writeFixture` 支持写任意相对路径到共享 fixture root（`test/e2e-comprehensive/orchestrator/fixtures.ts:8`、`test/e2e-comprehensive/orchestrator/fixtures.ts:15`）。
home root 相关要谨慎：同一个 `client-a` 容器共享 `$HOME` 和同一个 device-id 文件（`test/e2e-comprehensive/agent/index.ts:180`、`test/e2e-comprehensive/agent/index.ts:183`、`test/e2e-comprehensive/agent/index.ts:298`），所以不能在两个并发 run 里同时给 `.claude/CLAUDE.md` 放不同内容并期待按 cwd 隔离。更合理的布局是 home 文件使用全局 marker，例如 `.claude/f4-home-marker.txt = HOME_SHARED_EXPECTED`、`.claude.json = {"f4":"HOME_JSON_SHARED_EXPECTED"}`，用来证明 session B 读取 home root 时没有返回 cwdA 的 project marker。若要专门诱发 FileAgent per-device cache 污染，应选择相同 relPath 但不同 root：例如 `home-claude/project-marker.txt` 与 `project-claude/project-marker.txt` 使用不同 marker，防止 scope/root 映射错乱。
编排伪流程如下：先 `mockAdmin.loadScript` 为两个 prompt 返回简单 final，确保 CC 启动后不长时间跑工具；记录 `baseline = last admin event id`；`Promise.all` 启动 `clients.runAsync("client-a", cwdA)` 与 `clients.runAsync("client-a", cwdB)`；等待两个 `pty.spawn.ready`，分别按 `expectedCwd` 拿到 `sessionIdA` 和 `sessionIdB`（helper 当前按 detail.cwd 过滤，`test/e2e-comprehensive/orchestrator/server-events.ts:356`、`test/e2e-comprehensive/orchestrator/server-events.ts:363`）。
随后执行 probes：对 A、B 分别 `cat $HOME/.claude/f4-home-marker.txt`、`cat $HOME/.claude.json`、`cat $cwd/.claude/project-marker.txt`、`cat $cwd/.claude/settings.local.json`、可选 `cat $cwd/CLAUDE.md` 或父级 `CLAUDE.md`。这些命令必须通过 `serverExec.run`，因为它在指定 session namespace 内 spawn（`test/e2e-comprehensive/orchestrator/server-events.ts:469`、`test/e2e-comprehensive/orchestrator/server-events.ts:476`）。每个 stdout 正断言本 marker，负断言不含对方 marker，例如 session B 的 project read 不含 `PROJECT_A_ONLY`。
事件断言分三组：第一组等待每个 session 的 `file-proxy.read.served` 或 `file-proxy.client.requested`，要求 `sessionId`、`root`、`relPath`、`clientCwd` 对齐。第二组查 `since=baseline` 后是否存在 `sessionId=sessionB` 且 `clientPath` 以 `cwdA` 开头的事件；数量必须为 0，反向同理。第三组检查 `config-preloader.plan` 或 `cwd-ancestor.plan`，要求 `ancestorDirs` 对 A 不包含 `cwdB`，对 B 不包含 `cwdA`。这比只检查 stdout 更强，因为 stdout 可能来自 daemon 本地 cache，而事件能说明具体 root/cwd 的服务路径。
收尾沿用现有 `killAndVerifyExited` 模式，因为 F2/F4 已有并发 async run 后 kill/wait 的污染防护（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:247`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:258`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1075`）。fixture cleanup 使用 `cleanupFixture(caseId)`（`test/e2e-comprehensive/orchestrator/fixtures.ts:27`），homeFixture 如有写入则应设置可控 cleanup；但同 client 并发共享 `$HOME`，应尽量减少 homeFixture 差异写入。

## 5. 预期风险 / 选型权衡
关于 daemon sideband sessionId：需求说 `shadow.served / write.served` 被 server 转录时 `sessionId === null`，但当前代码不是这样。`handleFuseLine` 对 `type:"event"` 使用 `this.adminEvents?.record(evt.kind, this.sessionId, evt.detail)`，所以顶层 sessionId 已绑定当前 FileProxyManager session（`server/src/file-proxy-manager.ts:1561`、`server/src/file-proxy-manager.ts:1577`）。因此本设计不选择“向 daemon 注入 sessionId 并由 daemon emit”，而是选择在 server 转录时补充 `clientCwd`；这样避免把 session 归属重复维护在 Python 侧。
关于单 case vs 多 case：单 case 的优势是能同时覆盖同一 race 窗口，因为两个 session、同 device、不同 cwd 的并发状态只需建立一次。现有 F4 P1 已用 `Promise.all` 同时启动两个 `clients.runAsync`（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1119`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1125`），沿用该模型能最大化触发共享 `FileAgent`、共享 manifest、per-session FUSE/runtime 的组合风险。代价是失败诊断更复杂，所以必须在 helper 里提供事件摘要和 unexpected event 列表。
关于扩展现有事件 vs 新增事件：fileProxy read/client/shadow 属于已有观测面，扩展 detail 比新增平行事件更直接，因为现有 helper 已围绕这些 kind 建立（`test/e2e-comprehensive/orchestrator/server-events.ts:158`、`test/e2e-comprehensive/orchestrator/server-events.ts:211`、`test/e2e-comprehensive/orchestrator/server-events.ts:307`）。但 ancestor plan 不是 read served 事件，应该新增 `config-preloader.plan` 或 `cwd-ancestor.plan`，否则把“计算计划”伪装成“文件服务”会混淆语义。`ConfigPreloader` 的 buildPrefetchItems 是唯一集中表达预热路径的位置（`server/src/config-preloader.ts:116`、`server/src/config-preloader.ts:131`）。
关于内容断言：stdout marker 是必要但不充分，event path/cwd 是必要补强。只看 stdout 可能无法区分“真的经 fileProxy/FUSE”与“某个本地路径已经存在”；只看 event 也可能无法证明返回内容没串台。两者组合才能避免 P0-B 提到的“测试通过但绕过守护路径”问题。现有 INF-11 注释已经强调 `serverExec.run` 是 honest 触发 namespace FUSE 的入口（`test/e2e-comprehensive/orchestrator/server-events.ts:447`、`test/e2e-comprehensive/orchestrator/server-events.ts:452`）。
关于 home root 隔离：同 client 并发两个 session 共享 `$HOME` 是客观事实，`homeFixture` 写入的是 agent 的 `HOME_DIR` 下路径（`test/e2e-comprehensive/agent/index.ts:211`、`test/e2e-comprehensive/agent/index.ts:218`）。因此不能设计“session A home marker 与 session B home marker 不同”作为 cwd 隔离断言；那会把共享 home 误当成 per-cwd home。更合理的是验证 home roots 不被 cwd/project marker 污染，以及相同 root/relPath 的 cache 命中没有错误地映射到 project cwd。
## 6. 守护意图自查
本设计不使用 client 本地读取作为主断言。所有内容读取都通过 `serverExec.run(sessionId, ...)` 在指定 namespace 内执行，而该 helper 的目的正是触发 namespace 内 FUSE read/write（`test/e2e-comprehensive/orchestrator/server-events.ts:447`、`test/e2e-comprehensive/orchestrator/server-events.ts:451`、`server/src/server.ts:554`）。因此它不会绕过 `FileProxyManager`、daemon、runtime bind mount 这些被守护路径。
本设计也不把 `pty.spawn.ready detail.cwd` 当作 fileProxy 隔离证明。`pty.spawn.ready` 只在 PTY helper spawn 成功时记录 `cwd`、`homeDir`、`pid`（`server/src/pty-session.ts:207`、`server/src/pty-session.ts:211`、`server/src/pty-session.ts:212`），现有 F4 P1 已经覆盖这一级（`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1161`、`test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1165`）。P2 必须进一步读取 FUSE 路径并检查 fileProxy events，否则仍会停留在 P1 的浅层覆盖。
本设计避免用 `/admin/cache` 作为主判断。`cacheAdmin.lookupEntry` 只查 server 端 manifest 摘要（`test/e2e-comprehensive/orchestrator/server-events.ts:75`、`test/e2e-comprehensive/orchestrator/server-events.ts:80`），不能证明 runtime read 使用了该内容。FileAgent/cache 污染的主风险发生在 `tryServeReadFromCache` 与 `FileAgent.read` 类似的 `(deviceId, scope, relPath)` 命中路径（`server/src/file-proxy-manager.ts:1227`、`server/src/file-agent/index.ts:107`），所以必须触发 namespace read 并观察 `servedFrom:"cache"` 或相关 client requested/read served event。
唯一当前无法完全 honest 覆盖的是 `cwd-ancestor-N` 的真实 FUSE read，因为 server 实际没有把 ancestor roots 放入 `this.roots`（`server/src/file-proxy-manager.ts:258`、`server/src/file-proxy-manager.ts:261`），但 bootstrap 和 daemon 又引用了该 root 形态（`server/src/claude-session-runtime.ts:348`、`server/src/fuse-host-script.ts:451`）。本设计对此不伪装：先用 compute/preload/mount-plan probe 守“计划不串台”，并明确若要守真实 ancestor file read，需要先补齐 root 映射或接受当前测试暴露产品缺口。
## 7. 对需求规格的反馈
第一，路径有误。需求写 `server/test/e2e-comprehensive/`，当前仓库实际是 `test/e2e-comprehensive/`；`rg` 在 `server/test/e2e-comprehensive` 下找不到目录，而现有 F4 位于 `test/e2e-comprehensive/orchestrator/phase-p1.test.ts:1098`。设计和后续 PR 应按实际路径落地。
第二，daemon sideband `sessionId === null` 的描述与当前代码不符。`AdminEvent` schema 确实允许 `sessionId: string | null`（`server/src/admin-events.ts:8`、`server/src/admin-events.ts:12`），但 `FileProxyManager.handleFuseLine` 转录 sideband event 时使用 `this.sessionId`（`server/src/file-proxy-manager.ts:1571`、`server/src/file-proxy-manager.ts:1577`）。所以这里不需要“修 null sessionId”，需要的是补 `clientCwd`、`clientPath/fusePath` 等隔离维度。
第三，`cwd-ancestor-N` 的规格假设与当前 server root 映射不一致。`computeAncestorChain`、ConfigPreloader、bootstrap 都存在 ancestor 逻辑（`server/src/path-utils.ts:19`、`server/src/config-preloader.ts:131`、`server/src/claude-session-runtime.ts:331`），daemon 也识别 `cwd-ancestor-*`（`server/src/fuse-host-script.ts:451`），但 `FileProxyManager.roots` 未包含任何 `cwd-ancestor-N`（`server/src/file-proxy-manager.ts:258`、`server/src/file-proxy-manager.ts:261`）。如果需求要求“读取 ancestor CLAUDE.md 必须真实经过 fileProxy”，那当前代码需要先补产品行为；如果红线禁止改核心逻辑，则 P2 只能守计算计划与不跨兄弟 cwd 的观测。
第四，`home-claude / home-claude-json` 的“不同 cwd 内容隔离”表述需要澄清。同一个 `client-a` 的两个并发 run 共享 `$HOME` 和 device-id（`test/e2e-comprehensive/agent/index.ts:180`、`test/e2e-comprehensive/agent/index.ts:298`），因此 home root 本身不是 per-cwd 内容空间。可守的是“home root 不被 project cwd 子树污染”以及“per-device cache 不因另一个 cwd 的访问返回错误内容”，但不能要求同一 device 同一 home 下两个 cwd 拥有不同 home root 内容。
第五，若必须覆盖 FileAgent 本体而不是 FileProxyManager 的共享 store 命中，需要额外设计 FileAgent probe。当前 FUSE IPC handler 注释说明它不强制走 `FileAgent.read`，只是 FileAgent 与 FileProxyManager 共享 `ClientCacheStore`（`server/src/file-proxy-manager.ts:101`、`server/src/file-proxy-manager.ts:103`、`server/src/file-proxy-manager.ts:106`）。因此通过 FUSE read 能守共享 manifest/blob 污染，但不能声称直接执行了 `FileAgent.read`；若需求坚持 b 的“FileAgent cache hit logic”字面含义，应新增只读 admin probe 调用 FileAgent 并记录结果，或把措辞改为“FileAgent/FileProxyManager 共享 per-device cache 污染”。
